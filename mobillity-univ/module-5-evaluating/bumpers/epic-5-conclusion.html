Congratulations on Earning Trust in Your Work

Well done

You have built the discipline that turns a generated analysis into a trustworthy one. You no longer take a clean run as proof — you check the logic, confirm the numbers, and have the LLM help audit its own work.

What You’ve Accomplished

Three layers of verification that you can now apply on their own or together.

  • You verify by hand, fast. You read code logic for the wrong column or join, spot-check rows against the source, and sanity-check whether a result is even plausible — catching mistakes a clean run hides.
  • You direct the LLM to write its own checks. You can describe what should be true and have the LLM write sanity-check code that re-runs every time the analysis does, turning a one-off review into a repeatable safeguard.
  • You use the LLM as an evaluator. You ask it to review its code against your question and to propose fast checks worth running — a reviewer framing that catches what the author pass missed.
  • You audit an analysis end-to-end. You combine human checks, sanity-check code, and LLM review into one systematic pass and capture it in an audit summary you can stand behind.

Real-World Impact

The analyst whose numbers hold up under questioning is the one whose work gets used. Being able to say not just “here is the result” but “here is how I know it is right” is what makes mobility analysis credible to the people who act on it.

Key insight

As the LLM gets better at generating analyses, your value moves from producing them to evaluating them. Skeptical verification, not generation, is the skill that lasts.

What’s Next: Telling the Story

A verified result still has to reach someone. The final module turns trustworthy numbers into a narrative — structuring findings with a framework like the Pyramid Principle, drafting audience-adapted communication with the LLM, and generating shareable presentations straight from your analysis.

Before You Continue

Make sure the core moves feel solid:

  • Can you name three quick human checks and the kind of error each one catches?
  • Can you describe an assumption in plain language and have the LLM write code that tests it?
  • Can you use the LLM as a reviewer of its own analysis without treating it as the last word?
  • Could you audit a complete analysis and summarise it in a finding-method-result-action table?
