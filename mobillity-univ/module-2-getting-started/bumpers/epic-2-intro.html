Welcome to the Module

You’ve spent a module learning how LLMs predict text, what tokens and context windows are, and where casual prompting breaks down. Useful knowledge — but knowledge is not skill. This module is where you stop talking about LLMs and start directing them to do real data work.

The shift is sharp. The LLM does not analyze your dataset directly; it writes Python code that does, and you run that code, read the output, and decide what to ask next.

What You’ll Learn

Across five short lectures, you will build the LLM-assisted code-generation workflow from first principles to a working notebook.

  • The workflow itself. Why LLMs are unusually reliable at writing code — code can be run and checked — and what the loop looks like in practice: direct, generate, run, interpret, iterate.
  • Loading data in Google Colab with Claude. Your first hands-on session — open a notebook, ask the model to load a real CSV, run what it produces, and read the first rows.
  • Inspecting a dataset with pandas. Direct the model to ask the right questions of your data — its shape, its types, its missing values, its categorical breakdowns — and learn what each answer actually reveals.
  • Choosing the right format for the job. Why xlsx, PDFs, and screenshots produce unreliable output, and what makes CSV and GeoJSON the formats that travel through an LLM intact.

Why LLM-Assisted Data Analysis Matters

Mobility data is exactly the kind of work this approach was built for. GTFS feeds, citizen feedback exports, and Eurostat tables arrive as files, not as conversations — you cannot paste them into a chat window and expect a defensible answer. What you can do is direct an LLM to write the Python that loads them, inspect what is actually there, and notice the moment a format choice is about to cost you accuracy.

Remember

The LLM writes the code; you decide what to ask and how to read the answer. Direct, generate, run, interpret, iterate — that is the loop the rest of the course runs on.

Ready to Begin?

The first lecture answers the question that frames the whole module: why ask a model that predicts text to write code at all? The answer is the foundation everything else you do here is built on.
